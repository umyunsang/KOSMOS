# Contract: Tool bridge protocol — `tool_call` ↔ `tool_result` round trip

**Status**: ACTIVATING (frames defined in Spec 032; emit/consume code added by this Epic)
**Direction**: bidirectional (backend emits `tool_call`, TUI emits `tool_result`)
**Pairs by**: `call_id` (ULID, monotonic per session)

## Sequence

```
        ┌──────────────────┐                       ┌────────────────┐
        │ TUI              │                       │ Backend        │
        │ (Bun, Ink)       │                       │ (Python stdio) │
        └─────────┬────────┘                       └───────┬────────┘
                  │  ChatRequestFrame                      │
                  │ ─────────────────────────────────────► │
                  │                                        │  K-EXAONE
                  │                                        │  function_call
                  │                                        │  decision
                  │  ToolCallFrame{call_id, name, args}    │
                  │ ◄───────────────────────────────────── │
                  │                                        │
        Tool.ts dispatch                                   │
        (mcp.ts → mcp_server.py)                           │
                  │                                        │  await pending_calls[call_id]
                  │  ToolResultFrame{call_id, envelope}    │
                  │ ─────────────────────────────────────► │
                  │                                        │  inject {role:"tool", ...}
                  │                                        │  into messages, continue
                  │  AssistantChunkFrame (text continued)  │
                  │ ◄───────────────────────────────────── │
                  │  AssistantChunkFrame{done=True}        │
                  │ ◄───────────────────────────────────── │
```

## `ToolCallFrame` (backend → TUI)

| Field | Type | Constraint | Source |
|---|---|---|---|
| envelope `role` | `"backend"` | E3 | New role-allow-list entry |
| envelope `correlation_id` | matches the originating `ChatRequestFrame` | trace correlation | |
| envelope `transaction_id` | UUIDv7 | required (E4) — tool dispatch is a stateful side-effect | |
| `kind` | `"tool_call"` | discriminator | |
| `call_id` | `str` (ULID) | per-turn unique, monotonic | Generated by backend at function_call decoded |
| `name` | `Literal["lookup","resolve_location","submit","subscribe","verify"]` | primitive name only; non-primitive tools nest under `lookup` per Spec 031 | From K-EXAONE function_call.name |
| `arguments` | `dict[str, object]` | resolved JSON | From accumulated `input_json_delta` partials |

**Emit timing**: When `LLMClient.stream` yields a `tool_use` content_block_stop event with `inputResolved`. Synthesis: backend collects partial JSON deltas across the stream, parses on stop, emits one `ToolCallFrame`.

**Multi-tool**: a single LLM turn MAY produce multiple `ToolCallFrame`s (concurrent function calls). Each gets its own `call_id`. Permission gauntlet evaluates each independently per Spec 033.

## `ToolResultFrame` (TUI → backend)

| Field | Type | Constraint | Source |
|---|---|---|---|
| envelope `role` | `"tui"` | E3 | |
| envelope `correlation_id` | matches the originating `ChatRequestFrame` | | |
| envelope `transaction_id` | matches the `ToolCallFrame.transaction_id` | mandatory (E4) | |
| `kind` | `"tool_result"` | discriminator | |
| `call_id` | `str` (ULID) | matches `ToolCallFrame.call_id` | |
| `envelope` (payload) | `ToolResultEnvelope` | discriminated union per Spec 031 | |

`ToolResultEnvelope.kind` ∈ {`lookup`, `resolve_location`, `submit`, `subscribe`, `verify`} mirrors the originating tool primitive.

**Emit timing**: TUI's `Tool.ts` finishes dispatch (success or fail), packages result in the primitive envelope, sends.

**Failure cases**:
- Tool throws during execution → TUI emits `ToolResultFrame` with `envelope.error` set per primitive contract.
- TUI's `mcp.ts` connection to `mcp_server.py` dies → TUI emits a synthetic `ErrorFrame` with `code="mcp_disconnect"` correlated by `correlation_id`; backend treats this as terminal-error for the turn.

## Backend pending-call tracking

```python
# stdio.py (sketch — full code in Phase D tasks)
pending_calls: dict[str, asyncio.Future[ToolResultFrame]] = {}

async def _handle_chat_request(frame: ChatRequestFrame) -> None:
    while True:
        async for stream_event in client.stream(messages, tools=tools):
            if stream_event.type == "content_block_stop" and stream_event.content_block.type == "tool_use":
                call_id = ulid.new()
                tool_call_frame = ToolCallFrame(call_id=call_id, ...)
                pending_calls[call_id] = loop.create_future()
                await write_frame(tool_call_frame)
        # All tool calls for this stream emitted. Now await results.
        if not pending_calls:
            break  # No tools, done streaming, terminal assistant_chunk emitted upstream.
        results = await asyncio.gather(*pending_calls.values(), timeout=120)
        # Inject tool results into messages, loop.
        for r in results:
            messages.append({"role": "tool", "content": r.envelope.json(), "tool_call_id": r.call_id, "name": ...})
        pending_calls.clear()
        # Re-stream with augmented messages.
```

`tool_result` consumer hook in `_dispatch_inbound_frame` resolves the future:

```python
async def _on_tool_result(frame: ToolResultFrame) -> None:
    fut = pending_calls.get(frame.call_id)
    if fut and not fut.done():
        fut.set_result(frame)
```

Timeout: the outer `asyncio.gather` has a 120 s timeout per turn (configurable via `KOSMOS_TOOL_RESULT_TIMEOUT_SECONDS`). On timeout, all pending futures are cancelled, a synthetic `tool_result` with `error_type="timeout"` is injected, and the turn proceeds to wrap up.

## Multi-turn ReAct loop

Within one `ChatRequestFrame` turn, the backend MAY call `client.stream` multiple times (each iteration of the ReAct loop). Each invocation gets its own GenAI span as a child of the same `kosmos.turn` per ADR-0004. The `ChatRequestFrame.messages` list is **not re-emitted by the TUI** between iterations — backend mutates a local copy.

This preserves CC's ReAct loop semantics inside a single TUI request without per-iteration round trips. The TUI sees only the final `assistant_chunk{done=True}` (or `error`) per turn, plus any `tool_call` / `tool_result` frames in between.

## Audit trail

Each `ToolCallFrame` and `ToolResultFrame` MUST be appended to the Spec 024 `ToolCallAuditRecord` chain via `kosmos.observability.event_logger`. Audit fields:

```
correlation_id, transaction_id, call_id, tool_id, decision, granted_at, completed_at, error_type
```

The audit ledger is the single source of truth for "did this tool actually run for this citizen this turn." TUI rendering is illustrative only.
